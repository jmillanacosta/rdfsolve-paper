\begin{abstract}

% Basic intro
Biomedical knowledge is increasingly encoded in structured, web-accessible
databases that can, in principle, be queried and combined automatically.
% (2) Detailed background — comprehensible to scientists in related disciplines
The life sciences linked open data (LSLOD) cloud comprises hundreds of such
databases expressed in the Resource Description Framework (RDF), yet each
resource adopts its own vocabulary, class hierarchy, and modelling conventions.
Because schemas are typically documented in publications rather than as
machine-readable artefacts, discovering what classes and properties a dataset
actually uses requires manual inspection and must be repeated whenever data are
updated.
% The general problem
Prior meta-analyses estimated vocabulary reuse across LSLOD at under
20\,\% and concluded the cloud is far less connected than commonly assumed, but
these analyses remain descriptive: overlap is diagnosed statistically
without constructing integration bridges or producing reusable schema artefacts.
% Main result
Here we present a methodology that automatically extracts approximate schemas
from SPARQL endpoints, aligns them through a combination of instance-level URI
probing and curated SSSOM mapping sets, and infers a connectivity graph revealing
which dataset can be joined.
% Comparison to / addition over prior knowledge
Our approach materializes the extracted mapping-enriched schemas in linked-data
standards including JSON-LD, SHACL, VoID, and LinkML, each carrying full provenance, making every artefact independently citable, and
exportable to downstream formats such as SQL tables and Pydantic APIs.
SSSOM and type-level mappings derived from shared instances from different namespaces expose hidden bridges
between resources sharing no surface-level vocabulary.
Inference over transitive, inverse, and generalised mapping chains further
extends connectivity beyond what any single mapping source provides.
% General context
Applied to 235 LSLOD datasets, the pipeline produces citable, machine-readable
schema artefacts and a navigable connectivity graph, lowering the barrier for
federated querying, cross-resource data integration, and longitudinal schema
monitoring.
% Broader perspective
By making structural overlap explicit and actionable, this work lays a
foundation for schema-aware query agents and systematic interoperability
assessment across the biomedical semantic web.
The methodology is released as an open-source Python package with a
containerized web interface and reproducible pipeline scripts.

\end{abstract}


\section{Introduction}

\subsection{Knowledge representation in the life sciences}

Knowledge bases store structured, curated knowledge about a
domain ~\cite{guarino_ontologies_nodate}.


The Resource Description Framework (RDF) models information as subject--predicate--object
triples where nodes are identified by Uniform Resource Identifiers
(URIs).%~\cite{specifications}.%
RDF Schema (RDFS) and the Web Ontology Language (OWL) layer class hierarchies,
property constraints, and logical definitions on top of this
model.%~\cite{goodsources}.%
The biomedical community was among the earliest adopters of Semantic Web
technologies for knowledge management and clinical decision
support%~\cite{goodsources}%
, and the resulting Life Sciences Linked Open Data
(LSLOD) cloud now spans hundreds of %% TODO: insert exact count
RDF resources ~\cite{kamdar_empirical_2021}.

RDF offers a uniform data model for heterogeneous domains.
The use of URIs also makes resources web-dereferenceable: each entity can
be looked up via standard HTTP.
Semantics encoded in RDFS and OWL enable software reasoners to verify
consistency and generate novel inferences from existing
assertions
%~\cite{goodsources}%
, and the linking model exposes links to related resources even when underlying schemas differ
%~\cite{goodsources}%
.

\subsection{The gap}

Despite these properties, the landscape of semantic web resources in RDF for the life 
sciences faces significant interoperability challenges.
There is semantic heterogeneity due to varying entity
notations, diverse schemas, and insufficient common vocabulary
reuse~\cite{kamdar_empirical_2021}.
%"To the point where..." %decide phrasing, without mappings between equivalent entities, the question of to which point these datasets can truly be considered linked ~\cite{goodsources}.
Analyzing complex RDF knowledge graphs requires a certain degree of computational skills
and deep knowledge of domain-specific terminologies%~\cite{goodsources}%
, and researchers often struggle to
find sources available and accessible on the Web%~\cite{goodsources}%
.
Recent work on zero-shot SPARQL generation shows that even LLM agents must
probe endpoint schemas interactively before they can construct correct
queries~\cite{walter_grasp_2025}, which points at the practical importance of
discoverable, machine-readable schema information.
TogoMCP~\cite{togomcp} reinforces this conclusion: it identifies concise,
dynamically delivered schema context like ShEx shapes, sample triples, and dataset
statistics packaged in a YAML descriptor as the most crucial component for accurate LLM-driven SPARQL generation, more valuable than
complex behavioural orchestration logic.
Moreover, generating correct queries for rare or unseen URIs (the out-of-vocabulary
problem) remains difficult without a machine-readable schema to ground
generation~\cite{Reyd et al}, which becomes an issue where RDF datasets attempt
%% TODO: replace \cite{goodsources} with proper Reyd et al. citation key
to reuse standard terms but end up representing them with
different notations or namespaces, e.g.\ varying URI patterns for the
same ChEBI identifier, which technically prevents automated integration ~\cite{hoyt_unifying_2022}.

Knowing which ontologies a dataset imports does not reveal its actual
structure: which classes are instantiated, which entailments appear,
how often a property appears, or where values are missing cannot be inferred from ontology declarations
alone.

This is why two datasets that (partially) reuse the same ontology can have very different practical
shapes while two datasets with disjoint vocabularies could have overlapping
patterns that remain invisible without mappings.
A meta-analysis of approximately 80 LSLOD sources estimated vocabulary
reuse at under 20\,\% and concluded that the cloud "is not actually as
densely connected as is often
indicated"~\cite{kamdar_empirical_2021}.

A further conceptual barrier stems from the evolution of schema
philosophies.
The Semantic Web community has traditionally treated schemas as
\emph{descriptive}, promoting flexibility, while the database community
regards them as \emph{prescriptive}, enforcing
validation~\cite{ahmetaj_common_2025}.
The development of RDF schema rule validation languages such as SHACL and
ShEx~Ahmetaj; Mihindukulasooriya~\cite{ahmetaj_common_2025,mihindukulasooriya_rdf_2018} % add their specifications and original citations!
reflects a convergence of these
views.


\subsection{Related work}

\paragraph*{Linked data profiling and RDF shape induction.}
Several profiling systems extract structural summaries from SPARQL
endpoints.
Loupe~\cite{mihindukulasooriya_loupe_nodate} mines class, property, and triple-pattern
statistics and presents them through an interactive web interface.
ABSTAT~\cite{spahiu_understanding_2024} computes ontology-driven linked data summaries
using triples of the form subject type, property and object type, and applies a
\emph{minimalization} step that prunes patterns whose types are
supertypes of more specific patterns already present, yielding more
concise profiles than raw pattern enumeration.
However, ABSTAT profiles are per-dataset and do not link patterns across
resources or produce reusable schema artefacts in standard constraint
languages.
ProLOD~\cite{abedjan_profiling_2014} profiles linked open data using statistical
and structural analysis.
SPORTAL~\cite{hasnain_sportal_2016} automatically generates VoID descriptions from
public SPARQL endpoints, providing dataset-level metadata including class
and property counts, but does not attempt schema-level pattern extraction
or cross-endpoint mapping.

Several methods have been developed to analyze and characterize RDF data,
each addressing a distinct slice of the schema discoverability problem.
IndeGx~\cite{maillot_indegx_2023} indexes dataset descriptions for
federated source selection using declarative SPARQL rules and
OWL~RL reasoning to reflexively characterize endpoint quality and
behaviour.
VoID-ext~\cite{goodsources} extends VoID with property and class partition
statistics. % TODO: find proper VoID-ext citation

Tools to extract schemas automatically exist, including
sheXer~\cite{fernandez-alvarez_shexer_2026}, VoID-generator~\cite{bolleman_void-generator_2015}, 
Loupe~\cite{loupe}, Kamdar et~al.~\cite{kamdar_empirical_2021}, but adoption among life sciences
dataset providers is slow.
Mihindukulasooriya et~al.~\cite{mihindukulasooriya_rdf_2018} use machine
learning (Random Forest classifiers trained on profiled
RDF data from Loupe) to automatically induce integrity constraints in
SHACL and ShEx, achieving up to 97\,\% precision in cardinality
estimation.
Automated extraction is also technically constrained by endpoint
timeouts and deeply nested constraint patterns.
VoID-ext service descriptions, where published,
can enable features like auto-completion~\cite{emonet_sparqleditor}, demonstrating the
practical value of machine-readable schema metadata.
QLever extends this idea engine-side: it precomputes per-subject predicate
patterns at index time to deliver context-sensitive SPARQL autocompletion
with faster responses even over very large graphs~\cite{bast_autocompletion_2022},
and more recently supports zero-shot SPARQL generation from natural language via
an LLM agent that probes the endpoint schema interactively~\cite{walter_grasp_2025}.
Still, most workflows, providers and endpoints do not explicitly share schema artifacts
for downstream, decentralized uses.

\paragraph*{LLM-driven query orchestration.}
TogoMCP~\cite{togomcp} recasts the LLM as a protocol-driven orchestrator
using the Model Context Protocol (MCP), separating entity resolution
(via REST APIs such as UniProt and TogoID) from schema-guided SPARQL
generation.
Each database is described by a manually curated MIE (Metadata--Interoperability--Exchange)
file, a concise YAML document containing ShEx shapes, sample triples, and statistics that
is injected into the LLM context at query time.
The approach demonstrates that structured schema context is the single most important factor
for correct SPARQL generation, but inherits two limitations: MIE files must be created
semi-automatically per database, requiring manual review, and the system operates on
individual endpoints without native cross-endpoint JOIN support, relying on TogoID for
sequential identifier conversion between databases.

\paragraph{Complementary schema description approaches.}
Other frameworks describe the contents of RDF databases using
alternative representations.
The DBCLS-developed rdf-config~\cite{kawashima_nbdc_2018} provides a
human-readable YAML schema representation, manually curated for
resources in the RDF Portal and used for finding links across the databases hosted
in the RDF Portal.
SPARQL query collections offer another useful complement for describing
RDF resources, illustrating possible uses for a database.

More recently, Tirlet et~al.~\cite{tirlet_modular_nodate} proposes treating SPARQL
queries as composable modules that can be chained to build more complex
analyses.

Achieving high performance for complex aggregate queries on large knowledge graphs
remains an open challenge~\cite{papadaki_brief_2023}.
On the shape extraction side, sheXer~\cite{fernandez-alvarez_shexer_2026} extracts ShEx shapes from
RDF data, and shex-consolidator~\cite{fernandez-alvarez_extracting_2024}
merges independently extracted shape schemas into unified descriptions.

SynopsViz~\cite{bikakis_rdfsynopsviz_2014} provides hierarchical visual 
exploration and multilevel charting of large RDF datasets through incremental

\paragraph*{Federated query planning and decentralized data access.}
Prior literature work reveals knowing \emph{which endpoints hold which data}
is a prerequisite for efficient federated execution regardless of the underlying access model.  %citations needed
Federated SPARQL execution requires source selection: determining which federation members
hold data relevant to each subquery.
FedQPL~\cite{cheng_fedqpl_2020} formalizes this as logical query plans over heterogeneous
RDF interfaces.
Link Traversal Query Processing (LTQP)~\cite{payne_link_2023} addresses this by
using structural assumptions (Type Indexes) to prune the
traversal space. %% NEED TO BE CLEAR, DESCRIBE HOW THIS IS A DIFFERENT FOCUS
LODsyndesis~\cite{LODsyndesis} takes a complementary approach:
pre-computed specialized indexes and lattice-based connectivity metrics
over hundreds of datasets enable global-scale knowledge services such as
finding the most connected datasets and measuring integration potential,
but the system operates on static pre-computed indexes rather than live
endpoint mining and does not produce reusable schema artefacts.



\paragraph*{LSLOD meta-analysis.}
Kamdar et al. ~\cite{kamdar_empirical_2021} 
conducted a comprehensive meta-analysis of the LSLOD cloud.
Their Extraction Algorithm enumerated classes and the properties connecting them 
for a given endpoint.
The extracted schema graph was used to analyze vocabulary reuse
(estimated at 19.86\,\%), detect communities of similar content via
GloVe word-embedding similarity and Louvain
clustering, and quantify inter- and intra-linking 
patterns.
Their finding that the LSLOD cloud is not actually as densely connected as is 
often indicated directly motivates the present work.
However, their approach remains diagnostic: it identifies latent overlap via 
community detection but does not materialize integration bridges or produce 
actionable schema artefacts (other than the pickle NetworkX archived in figshare).
The authors call for top-down approaches to monitor source accessibility and 
enforce vocabulary reuse standards~\cite{kamdar_empirical_2021}.

\paragraph*{Identifier normalization and knowledge graph assembly.}
Babel \and ORION~\cite{carter_i_nodate} take a top-down approach to biomedical
data integration: Babel normalizes identifiers across databases to a
common namespace, and ORION assembles heterogeneous data into a unified
knowledge graph conforming to the Biolink Model.
This strategy yields a directly queryable integrated resource but requires
community-maintained ontologies and curated transformation rules for each
source, a design that trades automation for semantic precision.
rdf-solve operates in the complementary bottom-up direction: rather than
prescribing a target model, it discovers the \emph{de facto} schemas that
sources already expose and maps them post hoc, preserving each dataset's
native representation.

Several observations emerge from this comparison.
First, most profiling tools (Loupe, ABSTAT, ProLOD, SPORTAL) produce
per-dataset summaries but do not link findings across resources;
rdf-solve's mapping layer and connectivity graph fill this gap.
Second, the LLM-driven systems (GRASP, TogoMCP) demonstrate the value
of machine-readable schema context for query generation, but operate on
single endpoints at query time; the persistent, multi-format artefacts
produced by rdf-solve could serve as upstream context for such agents
across the entire LSLOD cloud.
Third, prior cloud-wide analyses (Kamdar \& Musen, LODsyndesis) quantify
fragmentation or connectivity diagnostically, without materializing
actionable integration bridges; rdf-solve produces typed mapping edges
and a connectivity graph that answer \emph{which endpoints can be joined},
not only \emph{which schemas are similar}.
Finally, top-down integration efforts (Babel \and ORION) achieve high
semantic precision at the cost of per-source curation; rdf-solve's
bottom-up, automated approach trades curation control for breadth of
coverage and repeatability.


\section{Methods}


\subsection{Scope}  % Needs rewrite, maybe this is more methods.

Formally, schema-based interoperability requires two capabilities.
The first is measuring \emph{consistency} within a dataset: how stable
is a structural pattern across instances.
Consistency scores, such as the coverage annotations introduced in
sheXer Fernández-Álvarez et~al.~\cite{fernandez-alvarez_automatic_2022}, capture how often a 
class--property--value-type pattern actually occurs.
The second is measuring \emph{alignment} across datasets: identifying
classes or properties in different graphs that correspond to each other.
This requires curated mappings or cross-references that link entities
across resources.

Several tools address parts of this space in isolation; however, no existing
system connects schema extraction, multi-format
serialization, mapping inference, and cross-resource connectivity
analysis to effectively expose links into a single pipeline.

rdf-solve addresses this gap.
Building on the class-then-property paradigm of prior schema profiling work,
it contributes:
\begin{enumerate}
\item When possible, local mining on QLever, which demands disk and RAM
      (Section~\ref{sec:mining}).  Because a local QLever server imposes no 
      network timeouts or rate limits, the pipeline first attempts a 
      \emph{one-shot} strategy: three unpaginated \texttt{SELECT DISTINCT} 
      queries (typed-object, literal, untyped-URI) that exhaust the full
      local index without pagination or batching.
      The multi-level fallback chain (bisection, adaptive pagination, 
      predicate-first decomposition) can be used as an alternative if
      any one-shot query fails.  Remote endpoints, where no data dump is
      available, always use the fallback chain;
\item Fault-tolerant remote endpoint mining with multi-level fallbacks
      (bisection, adaptive pagination, predicate-first decomposition)
      that attempts to exhaust all possibilities before
      silently discarding classes on failure (Section~\ref{sec:mining});
\item a configurable treatment of untyped URI patterns: the default
      strategy records them as bare predicates, while an alternative
      strategy (\texttt{--untyped-as-classes}) promotes the object
      URIs to candidate classes, enabling a second pass over the same
      indexed data that reflects a different modelling assumption
      (Section~\ref{sec:mining});
\item a mapping layer that aggregates instance-based URI probing,
      curated SSSOM mapping sets, and SeMRA identifier mappings into a
      unified type-level correspondence graph, with inference expansion
      via transitivity, inversion, and generalisation
      (Section~\ref{sec:methods-mapping});
\item serialization of every extraction step in standard linked data
      formats (JSON-LD, VoID, SHACL, LinkML) with full provenance
      metadata, including author attribution,  timing, and
      tool version, so that each artefact is independently citable; and
\item a connectivity graph that identifies which endpoint pairs can be
      concretely joined via shared ontology classes and inferred
      mappings
      (Section~\ref{sec:results-connectivity}).
\end{enumerate}
The tool is open-source, CLI-driven, and ships as a containerized web
application.

\subsection{Schema Mining}
\label{sec:mining}

rdf-solve approximates the schema of a SPARQL endpoint by mining
co-occurrence patterns from instance data, but minimizing timeout and service cost-related incompleteness
of the mined results by prioritizing local mining and, if performingit on a remote endpoint:


(i)~configurable class batching via
SPARQL \texttt{VALUES} bindings.
(ii)~a multi-level fallback chain (batch bisection, pagination with adaptive
page sizing, and predicate-first decomposition) that degrades
gracefully instead of discarding classes on failure; and
(iii)~separation of pattern discovery from counting, ensuring that a
count-query failure does not lose already-discovered patterns.

The extracted patterns of $(class_{subject}, p, class_{object})$ are closely related to those in related work ~\cite{spahiu_understanding_2024} ~\cite{mihindukulasooriya_loupe_nodate} ~\cite{abedjan_profiling_2014}~\cite{kamdar_empirical_2021}~\cite{bolleman_void-generator_2015}~\cite{fernandez-alvarez_shexer_2026}~\cite{hasnain_sportal_2016}

ABSTAT~\cite{spahiu_understanding_2024} additionally applies a minimalization step that prunes patterns
whose subject or object types are supertypes of more specific patterns
already present, reducing profile size at the cost of discarding
supertype-level information. Tools like VoID generator~\cite{bolleman_void-generator_2015} or shexer~\cite{fernandez-alvarez_shexer_2026} do not apply such pruning, retaining all observed patterns.

Likewise, rdf-solve retains all observed patterns.
Crucially, all of the above remain per-dataset tools; rdf-solve feeds
its patterns into a cross-resource mapping and connectivity layer that
links findings across endpoints.


Three pattern types are recorded: typed-object patterns $(c, p, o)$
where $o$ is the \texttt{rdf:type} of the object URI; literal
patterns $(c, p, \mathit{dt})$ where $\mathit{dt}$ is the object
datatype; and untyped-URI patterns $(c, p)$ where no
\texttt{rdf:type} was retrievable for the object under the current
graph scope and inference presented by the endpoint.

The treatment of the third pattern type is configurable.
In the default strategy, untyped-URI patterns are retained as bare
predicate links whose object type is unknown.
An alternative strategy promotes those object URIs to candidate
classes, effectively treating the absence of an explicit
\texttt{rdf:type} assertion as a modelling gap rather than an absence
of class membership.

Both strategies can be applied to the same indexed data in
consecutive passes, yielding two schema variants for the same source
that can be compared or combined.
This is particularly relevant for local mining (Section~\ref{sec:local-mining}),
where re-running the algorithm over an already-indexed QLever instance
adds negligible load.

Class and property labels are resolved from a priority-ordered label
vocabulary: \texttt{rdfs:label} is preferred, followed by
\texttt{skos:prefLabel}, \texttt{dc:title}, \texttt{IAO\_0000118}
(alternative term), and \texttt{skos:altLabel}.

Falling back to the local URI fragment is a last resort.
The multi-vocabulary lookup is relevant for endpoints that use
OBO-style annotation properties rather than RDFS labels.

\paragraph*{Notation.}
$Q(B, G)$ denotes query template $Q$ evaluated for class batch $B$
restricted to the named graphs in $G$, a set of zero or more URIs
supplied via \texttt{--graph-uri}.
When $G = \varnothing$ no \texttt{GRAPH} clause is added and the
query runs over the endpoint's default dataset.
When $|G| = 1$ a single \texttt{GRAPH <uri> \{ \ldots \}} clause
wraps the pattern; when $|G| > 1$ a
\texttt{VALUES (?g) \{ \ldots \} GRAPH ?g \{ \ldots \}} block
iterates over each named graph.
In all cases the scope is a query-level restriction; endpoint dataset
configuration may override or ignore it.
$\top$ / $\bot$ denote success / failure (timeout or cost-limit).
$B_{[:m]}$ and $B_{[m:]}$ are the two halves of $B$ after splitting
at $m=\lfloor|B|/2\rfloor$; bisecting reduces the number of classes
in the \texttt{VALUES} binding, narrowing the class-filter scope and
isolating expensive classes into separate sub-queries.
$\ell_{\min}$: minimum page size ($60\,\%$ of the initial
\texttt{--chunk-size}); pagination is abandoned after three
consecutive failed reductions at the same offset.
$Q_C$: \texttt{SELECT DISTINCT ?c WHERE \{?s a ?c\}} (type
enumeration).
$Q_p(c)$: \texttt{SELECT DISTINCT ?p WHERE \{?s a <$c$>; ?p ?o\}}
(predicate enumeration for class $c$).
$Q_\mathrm{to}^p(c,p)$: $Q_\mathrm{to}$ restricted to predicate $p$
for class $c$.
$t \models (s,p,o)$: triple $t=(s',p',o')$ matches pattern $(s,p,o)$
operationally if the endpoint's evaluation of
\texttt{?s a <$s$>; <$p$> ?o; ?o a <$o$>} (or the literal/datatype
variant) binds $t$; this depends on the endpoint's graph scope,
asserted \texttt{rdf:type} triples, and any active inference regime.
$n_{(s,p,o)} = |\{t \mid t \models (s,p,o)\}|$ is the co-occurrence
count as returned by the endpoint's \texttt{COUNT(*)} aggregate.
$Q_\mathrm{to}^{cnt}$, $Q_\mathrm{lit}^{cnt}$,
$Q_\mathrm{uri}^{cnt}$: the corresponding
\texttt{SELECT \ldots COUNT(*) \ldots GROUP BY} variants of
$Q_\mathrm{to}$, $Q_\mathrm{lit}$, $Q_\mathrm{uri}$; they accept
the same \texttt{VALUES} batching and fallback chain.

\paragraph*{Procedure.}
\textsc{Query}$(B, Q, E, G)$ sends $Q(B, G)$ to endpoint $E$ as a
single \texttt{SELECT DISTINCT} query.
On $\top$ it returns the results immediately.
On $\bot$, if $|B| > 1$ the batch is bisected and each half retried
recursively: reducing the \texttt{VALUES} binding narrows the set of
classes evaluated per sub-query, so that a class whose query exceeds
the endpoint's cost limit does not cause the remaining classes in the
same batch to be skipped.

For single-class batches ($|B|=1$) the class filter is already
minimal, so the fallback is pagination (\texttt{LIMIT $\ell$} /
\texttt{OFFSET $k\ell$}, $k = 0, 1, 2, \ldots$).

If a page times out, $\ell$ is reduced by a configurable shrink factor
(default $0.85$, i.e.\ a ${\sim}15\,\%$ reduction per attempt) and the
same page is retried after a cooldown pause; the chunk size is clamped
to a floor of $60\,\%$ of the original value and at most three
consecutive reductions are attempted per offset.
If all reductions fail, pagination is abandoned and, for
$Q_\mathrm{to}$, a predicate-first decomposition is attempted:
$Q_p(B[0])$ enumerates predicates $P$ and
$Q_\mathrm{to}^p(B[0],p)$ is called once per $p \in P$; if $Q_p$
itself fails, the class contributes no typed-object patterns.
Any results accumulated before pagination was abandoned are retained.
If all strategies fail, \textsc{Query} returns $\emptyset$.

Phase~1 enumerates the asserted instantiated types $C$ visible
under~$G$.
When a \texttt{class\_chunk\_size} is configured, the type-enumeration
query is paginated with \texttt{LIMIT}/\texttt{OFFSET} steps using the
adaptive page-sizing mechanism (the page size is reduced on timeout, as
described above); otherwise $Q_C$ is sent as a single unpaginated query.
Phase~1 does \emph{not} use the bisect/decompose fallback chain: the
type-enumeration query is a lightweight single-hop scan
(\texttt{?s a ?class}) that does not involve the multi-way joins that
motivate bisection in Phase~2.
Phase~2 partitions $C$ into batches $B$ of configurable size and
calls \textsc{Query} three times per batch ($Q_\mathrm{to}$,
$Q_\mathrm{lit}$, and $Q_\mathrm{uri}$) accumulating patterns into
$S$.
If $G \neq \varnothing$ and Phase~2 yields $S = \emptyset$ despite
$C \neq \emptyset$, an \emph{ontology-graph fallback} is triggered:
Phase~2 is re-run with $G' = \varnothing$, because the named graph
may contain only class definitions while instances reside in the
default graph. % Example, when they have well-defined ontology vs data graphs, happens with SIB a lot
Optionally (enabled by default, disabled with \texttt{--no-counts}),
a counting phase follows: the unique subject classes in $S$ are
re-batched identically to Phase~2 and each batch is submitted as a
grouped \texttt{SELECT \ldots COUNT(*) \ldots GROUP BY} query through
\textsc{Query}, so count queries benefit from the same bisect and
pagination fallbacks as pattern queries.
If a count query fails for a particular class (after all fallbacks),
the corresponding patterns are retained in $S$ with no count.

\paragraph*{Execution strategy selection and schema completeness.}
For local QLever endpoints the preferred execution path is the one-shot
strategy: a single unpaginated \texttt{SELECT DISTINCT} issued before any
fallback is attempted.
If it succeeds, the schema is complete in one round-trip and the fallback
chain is never entered.
Only when the one-shot query fails does the pipeline activate the
multi-level fallback chain (Algorithm~\ref{alg:mine}).
Remote endpoints always begin with the fallback chain, as the endpoint's
timeout and cost-limit behaviour is unknown.

When a source has been mined under multiple execution strategies
(e.g.\ default and \texttt{untyped-as-classes}, or local and remote),
the pipeline selects the most complete result for downstream analysis.
Completeness is assessed by two criteria applied in order: first,
whether the counting phase succeeded for all classes (i.e.\ all patterns
carry co-occurrence counts rather than null); second, total pattern count.
The run with full count coverage is preferred; among runs with equal
coverage, the one with the larger pattern set is retained.
This selection is recorded in the provenance block of the emitted schema,
allowing the choice to be audited and overridden.  % Clarify this is for the "production" schemas emitted for downstream users

Algorithm~\ref{alg:mine} summarizes the full procedure.

\begin{algorithm}[H]
\caption{Schema pattern mining}
\label{alg:mine}
\begin{algorithmic}[1]
  \scriptsize
\Require endpoint $E$; graph scope $G$
\Ensure $S$ (observed co-occurrence patterns); $\emptyset$ for a
        batch may mean failure or genuinely no patterns;
        incompleteness is logged but not encoded in $S$
\State $S \leftarrow \emptyset$
\Statex
\textbf{Phase 1 - class discovery}
\State $C \leftarrow$ \Call{Query}{$\varnothing$, $Q_C$, $E$, $G$}
       \Comment{paginated with adaptive page sizing if \texttt{class\_chunk\_size} set;
                single unpaginated query otherwise}
\Statex
\textbf{Phase 2 - pattern mining}
\For{each batch $B \subseteq C$}
  \State $S \leftarrow S \cup$ \Call{Query}{$B$, $Q_\mathrm{to}$, $E$, $G$}
         \Comment{typed-object patterns}
  \State $S \leftarrow S \cup$ \Call{Query}{$B$, $Q_\mathrm{lit}$, $E$, $G$}
         \Comment{literal patterns}
  \State $S \leftarrow S \cup$ \Call{Query}{$B$, $Q_\mathrm{uri}$, $E$, $G$}
         \Comment{untyped-URI patterns}
\EndFor
\If{$S = \emptyset$ \textbf{and} $C \neq \emptyset$ \textbf{and} $G \neq \varnothing$}
       \Comment{ontology-graph fallback}
  \State re-run Phase~2 with $G' = \varnothing$
\EndIf
\Statex
\textbf{Counting phase} (optional; skipped with \texttt{--no-counts})
\State $N \leftarrow \emptyset$
\For{each batch $B \subseteq C_S$}
        \Comment{$C_S$: unique subject classes in $S$}
  \State $N \leftarrow N \cup$ \Call{Query}{$B$, $Q_\mathrm{to}^{cnt}$, $E$, $G$}
         \Comment{typed-object counts}
  \State $N \leftarrow N \cup$ \Call{Query}{$B$, $Q_\mathrm{lit}^{cnt}$, $E$, $G$}
         \Comment{literal counts}
  \State $N \leftarrow N \cup$ \Call{Query}{$B$, $Q_\mathrm{uri}^{cnt}$, $E$, $G$}
         \Comment{untyped-URI counts}
\EndFor
\State $S \leftarrow \{\,(s,p,o,n_{(s,p,o)}) \mid (s,p,o)\in S\,\}$
        \Comment{merge; $n$ = \texttt{null} when no count retrieved}
\State \Return $S$
\Statex
\Function{Query}{$B$, $Q$, $E$, $G$}
  \If{$Q(B,G) = \top$} \Return results \EndIf
        \Comment{single \texttt{SELECT DISTINCT}}
  \If{$|B| > 1$} \Comment{bisect: isolates expensive classes into separate sub-queries}
    \State \Return \Call{Query}{$B_{[:m]}$, $Q$, $E$, $G$}
            $\cup$ \Call{Query}{$B_{[m:]}$, $Q$, $E$, $G$}
  \EndIf
  \State $R \leftarrow \emptyset$;\ $k \leftarrow 0$;\ $r \leftarrow 0$
        \Comment{$r$: consecutive shrink attempts at this offset}
  \Repeat \Comment{paginate single-class result set (\texttt{SELECT DISTINCT})}
    \If{page $Q(B,G,\ell,k) = \top$}
      \State $R \leftarrow R \cup \text{page results}$;\ $k \leftarrow k+1$;\ $r \leftarrow 0$
    \Else
      \State $\ell \leftarrow \max(\lfloor 0.85\,\ell \rfloor,\;\ell_{\min})$;\
             $r \leftarrow r + 1$;\ \textbf{retry page} $k$
      \If{$r > 3$ \textbf{or} $\ell = \ell_{\min}$ unchanged} \textbf{goto} \textit{decompose} \EndIf
    \EndIf
  \Until{empty page received}
  \State \Return $R$ \Comment{partial results are retained even if pagination is later abandoned}
  \Statex
  \State \textit{decompose:}
  \If{$Q = Q_\mathrm{to}$} \Comment{predicate-first decomposition (last resort)}
    \State $P \leftarrow$ \Call{Query}{$\{B[0]\}$, $Q_p(B[0])$, $E$, $G$}
    \State \Return $\displaystyle\bigcup_{p \in P}$
            \Call{Query}{$\{B[0]\}$, $Q_\mathrm{to}^p(B[0],p)$, $E$, $G$}
  \EndIf
  \State \Return $\emptyset$ \Comment{strategies exhausted}
\EndFunction
\end{algorithmic}
\end{algorithm}

\subsection{Semantic mapping layer}
\label{sec:methods-mapping}

The mapping layer aggregates cross-resource correspondences from three
complementary sources:

\begin{enumerate}
\item \textbf{Instance-based URI probing.}
      For each bioregistry~\cite{bioregistry} prefix (e.g.\ \texttt{ensembl},
      \texttt{chebi}), all registered URI format prefixes are resolved via
      \texttt{bioregistry.get\_uri\_prefixes}.
      Each format is then tested against every data source via a
      \texttt{STRSTARTS}-based \texttt{SELECT DISTINCT ?c} query that
      retrieves the \texttt{rdf:type} classes whose instances carry URIs
      matching that prefix.
      Pairwise mapping edges are generated between every two \emph{distinct}
      classes that both matched the resource, including two classes within
      the \emph{same} dataset (e.g.\ \texttt{Gene} and
      \texttt{GeneAnnotation} in the same endpoint both having Ensembl
      instance URIs).
      The default predicate is \texttt{skos:narrowMatch}, reflecting that
      URI-prefix co-occurrence is necessary but not sufficient evidence for
      semantic equivalence.

\item \textbf{SSSOM imports.}
      Curated mapping sets published in the Simple Standard for Sharing
      Ontology Mappings (SSSOM) format~\cite{goodsources} are downloaded and
      converted to the internal mapping representation.
      Each \texttt{.sssom.tsv} file is parsed including its YAML
      front-matter; predicate URIs are resolved first through the file's own
      \texttt{curie\_map} and then through a built-in fallback table for
      standard SKOS and OWL predicates.
      SeMRA~\cite{hoyt_assembly_2025} provides systematically assembled
      identifier-level mappings covering prefix combinations (e.g.\
      NCIT--ChEBI) not visible from endpoint probing alone.
      These are imported via a bidirectional converter between rdf-solve's
      mapping edge representation and SeMRA's native \texttt{Mapping}
      objects, preserving evidence metadata.
\end{enumerate}

All three sources produce edges in the same internal format
(\texttt{MappingEdge}: source class, target class, predicate, source and
target dataset and endpoint), enabling uniform downstream processing.

The aggregated edges are then generalised to type-level correspondences
via the class-index lookup and type-pair derivation described below.
Inference over the combined mapping graph subsequently propagates new
links by applying transitivity, inversion, and optionally generalisation,
using the SeMRA inference engine~\cite{hoyt_assembly_2025}
(Section~\ref{sec:inference}).

All mappings carry provenance metadata including evidence chains and
justification predicates from the SSSOM and semapv vocabularies.

\paragraph*{Type lookup.}
\label{sec:class-index}

For every entity $e$ in the mapping set $M$, the pipeline first expands $e$ to all
registered URI prefix forms for the same identifier using bioregistry
\cite{bioregistry} (e.g.\ \texttt{identifiers.org}, \texttt{purl.obolibrary.org},
and OBO PURL forms for the same entity), ensuring that the query covers every
representation that may appear in $G$.
Entities are then looked up in batches via a single SPARQL query that retrieves, for
each entity, the named graphs in which it appears and the \texttt{rdf:type} values
asserted in each graph:
\begin{lstlisting}[language=SPARQL, numbers=none]
SELECT ?e ?g ?type WHERE {
  VALUES ?e { <iri_1> <iri_2> ... }
  GRAPH ?g { ?e a ?type }
}
\end{lstlisting}
Results from all URI forms for the same entity are merged back under its canonical
identifier.
The resulting lookup table is optionally cached so that repeated runs do not
re-query the endpoint.

\paragraph*{Type-pair derivation.}

Algorithm~\ref{alg:classderive} formalises the derivation step.

\begin{algorithm}[H]
\caption{Type-level mapping derivation from instance evidence}
\label{alg:classderive}
\begin{algorithmic}[1]
  \scriptsize
\Require mapping set $M$ (instance edges $(s,p,o)$);
         type lookup $\mathcal{I}: e \mapsto \{\text{graph} \mapsto [\text{type},\ldots]\}$;
         thresholds $n_{\min}$, $\tau$
\Ensure type-pair list $\mathcal{P}$, sorted by confidence
\State $A \leftarrow$ empty counter \Comment{key: $(t_s, t_o)$; value: supporting instance count}
\For{each $(s, p, o) \in M$}
  \For{each type $t_s$ of $s$ in $\mathcal{I}$}\ \For{each type $t_o$ of $o$ in $\mathcal{I}$}
    \State $A[t_s, t_o] \mathrel{+}= 1$
  \EndFor\ \EndFor
\EndFor
\State $\mathcal{P} \leftarrow \bigl\{\,(t_s, t_o, A[t_s,t_o])\;\bigm|\;
        A[t_s,t_o] \geq n_{\min}
        \;\text{and}\;
        \mathrm{conf}(A[t_s,t_o]) \geq \tau\,\bigr\}$
\State \Return $\mathcal{P}$ \Comment{$\mathrm{conf}(n)=\min(1,\log_2(1+n)/10)$. Placeholder,
        see Section~\ref{sec:mapping-quality-disc}}
\end{algorithmic}
\end{algorithm}

The output type pairs with supporting instance counts and confidence scores are
serialised as a mapping artefact with full provenance metadata alongside a run report
recording lookup cost, the fraction of entities successfully resolved, and the
distribution of pair counts after filtering.
An optional enrichment step annotates the original instance mapping files in place
with the resolved type information, allowing downstream inspection of which ontology
class types were recovered for each entity.

\paragraph*{Inference expansion.}
\label{sec:inference}

After type-pair derivation, inference over the combined mapping graph
propagates new links using the SeMRA inference
engine~\cite{hoyt_assembly_2025}.
Three operations are applied in sequence:
(i)~\emph{inversion}: every mapping is made symmetric by generating
its reverse;
(ii)~\emph{transitivity}: transitive chains are inferred up to a
configurable maximum chain length (default~3);
(iii)~\emph{generalisation} (optional): broader/narrower
relationships are propagated to derive additional correspondences.
After inference, duplicate mappings are deduplicated via evidence
assembly, merging the evidence chains from all sources that produced
the same mapping.
The output is a single consolidated mapping artefact with provenance
metadata recording which inference operations were applied and which
source files contributed.

\subsection{Standard serializations}

Extracted schemas are serialised in several standard formats.
JSON-LD documents include a provenance block recording the extraction strategy,
endpoint URL, generation timestamp, and pattern count, making each schema
self-describing and ready for downstream use without requiring external metadata.
VoID-ext Turtle files describe class and property partitions following the extended
VoID vocabulary~\cite{goodsources}, generated directly from the internal schema model.
SHACL shapes are produced through a two-step conversion: the JSON-LD schema is first
translated to a LinkML schema, which is then passed through the LinkML
\texttt{ShaclGenerator}; the resulting shapes can be applied directly for data
validation or repair.
JSON-LD, SHACL, and VoID artefacts are compatible with standard RDF tooling including
rdflib and Apache Jena.
The LinkML intermediate additionally enables downstream exports such as Pydantic
Python APIs and SQL tables.  %% TODO: these exports are planned, not yet implemented.

Provenance blocks additionally record the tool version, Python version, 
duration, machine fingerprint, and, where supplied, the ORCID-bearing author list of the
person or workflow that triggered the extraction.
This allows mining runs to be cited and traced to specific pipeline invocations.
All schema and VoID artifacts are assigned canonical URIs under a shared base, so
cross-referencing between a report, its VoID catalog, and its LinkML schema is
machine-readable without extra configuration.

The web backend persists all artefact types (mined schemas, VoID catalogs, LinkML
schemas, and mining reports) in a structured database, and exposes them through
dedicated API endpoints (\texttt{/api/reports}, \texttt{/api/void},
\texttt{/api/linkml}, etc.).
This decouples downstream consumers from the filesystem layout of the pipeline output
and allows artefacts to be queried, filtered by dataset or strategy, and downloaded
in their native formats (Turtle, YAML) directly via HTTP.

\subsection{Implementation and reproducibility}
\label{sec:implementation}

The methodology is packaged as two Docker images.

The \emph{pipeline script} bundles QLever, the format converters, and the
extraction and mapping tools; it is the environment used for all experiments
reported in this paper.
This pipeline drives the complete workflow:
running remote VoID discovery, optionally mining remote endpoints directly,
generating per-source QLever configurations,
downloading and indexing each dataset, mining schemas locally, executing the
mapping and inference stages, and collecting results.
The mapping stages comprise six dedicated pipeline steps: seeding SSSOM
mapping bundles; importing SeMRA identifier mappings; running instance-based
URI probing against each data source; restarting all locally indexed
datasets as a unified LSLOD endpoint; building a class index by expanding
every entity IRI to all registered bioregistry URI forms and querying the
LSLOD endpoint for \texttt{rdf:type} assertions per named graph; and
deriving type-pair correspondences with confidence scores from the
aggregated instance evidence (Algorithm~\ref{alg:classderive}).
An optional enrichment pass annotates each source mapping file in place
with the resolved type information.
A separate inference step then applies transitivity, inversion, and
generalisation over the combined mapping graph.


A separate \emph{web application container} packages a Flask backend with a
SQLite store for the extracted schemas and mappings, together with a TypeScript
frontend for interactive exploration: dataset and mapping-source selection,
diagram-based schema visualization, a SHACL pattern browser and editor, and an IRI
resolver-based query generator.
The web container is not required to reproduce the results; it serves as an
interface for navigating the outputs produced by the pipeline. % A lot TODO

\paragraph{Local mining}
\label{sec:local-mining}

Remote SPARQL mining depends on endpoint availability, rate limits, and timeout behavior,
all of which vary across engines and time. To decouple extraction from these constraints,
rdf-solve supports a local mining mode that operates on downloaded data dumps.

For each source that publishes downloadable data (N-Quads, RDF/XML, or compressed archives),
the pipeline generates a QLever configuration file containing the appropriate download and
conversion commands. QLever~\cite{bast_qlever_2017} indexes the data locally and exposes a
SPARQL endpoint, against which the same mining algorithm (Algorithm~\ref{alg:mine}) is applied.

Because the local engine runs without network overhead, timeouts, or rate limits, mining
completes faster and with fewer fallback activations.

The absence of network constraints makes a qualitatively different execution mode practical.
For local QLever instances the pipeline issues three \emph{single unpaginated}
\texttt{SELECT DISTINCT} queries (one each for typed-object, literal, and
untyped-URI patterns) that attempt to retrieve the complete schema from the
full index without pagination or batching.
When all three succeed, which is reliably the case for
all but the very largest datasets, % This is what I think, wait to run the analyses in the server to re-write this
no fallback logic is needed and the schema is
complete in three round trips.

This execution path is \emph{only} available for local mining; remote endpoints always
begin directly with the fallback chain, since the endpoint's timeout
and rate-limit behaviour is unknown.

This local mode complements remote mining. Sources that offer both a SPARQL endpoint and
data dumps can be mined both ways, providing a cross-validation baseline for extraction
completeness. Sources with unstable or slow endpoints can be mined locally without depending
on their availability, and some sources with unmaintained endpoints need to be mined locally.

The pipeline supports executing the default and the untyped-as-classes extraction strategies
consecutively for each source without re-downloading or re-indexing.
The resulting pair of schema variants for each dataset allows downstream analysis to assess
how much additional structure is recovered when untyped URI patterns are promoted to classes,
and whether those candidate classes correspond to known ontology terms.

\section{Discussion}

\subsection{Positioning relative to SPARQL engine advances}

Recent engineering work on QLever addresses the \emph{engine} side of
the scalability problem: block-based storage, precomputed predicate
patterns, and autocompletion over billion-triple
graphs~\cite{bast_qlever_2017,bast_autocompletion_2022}.
GRASP shows that an LLM agent can generate correct SPARQL for a
single, well-maintained endpoint by probing its schema
interactively~\cite{walter_grasp_2025}.
These are substantial technical advances.

\paragraph*{Cross-endpoint against single-endpoint.}
Federated queries or dataset selection across the LSLOD cloud require
prior knowledge of which endpoints share compatible schemas.
The connectivity graph produced here provides that prior, though its
quality depends directly on the completeness of the mined schemas and
the precision of the imported mapping sets, both of which carry
known limitations (Section~\ref{sec:mining-discussion}).

\paragraph*{Persistent, citable artefacts versus on-the-fly discovery.}
GRASP discovers schema information at query time.
The artefacts produced here (VoID catalogs, SHACL shapes, LinkML
schemas) are static snapshots with provenance metadata.

\paragraph*{Open science outputs for under-resourced endpoints.}
The resources that QLever autocompletion and LLM approaches perform
best on tend to be large, consistently available, well-maintained
endpoints (Wikidata, UniProt).
Much of the LSLOD cloud is: small-to-medium databases
maintained by small groups, with intermittent support, no VoID-ext or other schema
descriptions, and no budget for infrastructure upgrades.
Publishing extracted schemas as linked data lowers the barrier for
tools and workflows that cannot query the source directly.
Whether this constitutes a meaningful contribution depends on adoption.

~\cite{kamdar_empirical_2021} concludes that the LSLOD
cloud "is not actually as densely connected as is often indicated",
with vocabulary reuse at under 20\,\% and many sources standing as
isolated RDF conversions.
Their community detection identified groups of semantically similar
schema elements across sources using word-embedding similarity,
revealing latent overlap invisible at the URI level. % But what didn't they do that I improve, and what exactly do I find, I can only write after running full analyses.
SynopsViz~\cite{bikakis_rdfsynopsviz_2014} and Sparklis~\cite{ferre_sparklis_2016} address
exploration and querying of individual sources but do not bridge the
gap across endpoints.
LODsyndesis~\cite{LODsyndesis} provides connectivity metrics across hundreds of datasets
but operates on pre-computed indexes rather than live endpoint
mining.

We materialize the connectivity that prior analyses only
detected statistically.
The connectivity graph (Section~\ref{sec:results-connectivity}) identifies endpoint pairs
that can be concretely joined via shared ontology classes and inferred
mappings, answering ``which endpoints can be federated?'' rather than ``which schemas are similar?''

Curated mapping sets imported further cover cases where namespace alignment alone is insufficient. % When I have results I can quantify this or qualify it

\subsection{Comparison with LLM-driven query orchestration}
\label{sec:togomcp-comparison}

TogoMCP~\cite{togomcp} demonstrates that structured schema context
is the decisive factor for accurate SPARQL generation by an LLM,
finding it more valuable than elaborate behavioural prompting.
Their MIE files, consisting of YAML documents bundling ShEx shapes, sample triples,
and basic statistics per database, serve a role analogous to
rdf-solve's mined schemas: both supply the structured descriptions that
endpoints themselves rarely publish.
The two systems are, however, complementary rather than competing,
differing in scope, automation, and downstream use.

\paragraph*{Automated extraction versus manual curation.}
Each MIE file is curated semi-automatically per database with
manual review~\cite{togomcp}, a process that scales well for the
approximately 70 resources in the RDF Portal but becomes a bottleneck
when extended to the full LSLOD cloud.
rdf-solve generates schema descriptions automatically through its
mining algorithm (Algorithm~\ref{alg:mine}), with the multi-level
fallback chain and one-shot local mining recovering patterns from
hundreds of endpoints without per-source engineering.
The trade-off is precision: manually reviewed ShEx shapes may be more
accurate for individual endpoints, while mined schemas trade curation
effort for breadth of coverage.
Because rdf-solve serialises its output as JSON-LD, LinkML, and SHACL,
the mined schemas could in principle be reformatted as MIE files,
supplying TogoMCP-style LLM context for databases that lack manually
curated descriptors.

\paragraph*{Cross-endpoint mappings and connectivity.}
TogoMCP operates on individual endpoints and bridges databases via
sequential identifier conversion through TogoID, without native
cross-endpoint JOIN support~\cite{togomcp}.
rdf-solve's mapping layer aggregates instance-based URI probing,
curated SSSOM mapping sets, and SeMRA identifier mappings into a
unified type-level correspondence graph
(Section~\ref{sec:methods-mapping}), with inference expansion via
transitivity, inversion, and generalisation.
The resulting connectivity graph directly answers which endpoint pairs
can be concretely joined, a capability that MIE-based approaches do
not address.

\paragraph*{Entity resolution.}
TogoMCP decouples entity resolution from SPARQL generation by
delegating identifier lookup to external REST APIs (UniProt, ChEMBL,
TogoID), a design that prevents the LLM from fabricating identifiers
within queries.
rdf-solve addresses a complementary aspect of the same problem:
its bioregistry-based URI expansion
(Section~\ref{sec:class-index}) resolves every entity IRI to all
registered URI forms before querying the LSLOD endpoint for
\texttt{rdf:type} assertions, ensuring that identifier notation
heterogeneity does not cause silent misses in the class index.

\paragraph*{MCP integration as future work.}
Exposing rdf-solve's mined schemas, mappings, and connectivity graph
as MCP tool resources would allow LLM agents to consume
these artefacts at query time in the same protocol-driven pattern
demonstrated by TogoMCP.
This would combine rdf-solve's automated, cross-endpoint schema
descriptions with the interactive query generation workflow that
TogoMCP has shown to be effective, without requiring per-database
manual curation of context files.

\subsection{Provenance and reproducibility}

Each mining run produces a self-contained report that records the tool
version, extraction strategy, endpoint URL, graph scope, 
duration, warnings found, and the identity of the person or automated workflow that
triggered it.
These reports are assigned canonical URIs and cross-referenced with
their corresponding VoID catalog, LinkML schema, and SHACL shapes,
so the full provenance chain of any artefact can be reconstructed
without consulting external logs.

Documenting the extraction strategy, including
whether counts were collected and whether the untyped-as-classes
variant was applied, means that two schemas generated from the
same endpoint under different assumptions are distinguishable
in the record.
This is a prerequisite for any meaningful longitudinal comparison or
method benchmarking: conclusions about schema evolution or strategy
differences require that the methodological parameters of each run
are unambiguously captured alongside the results.

\subsection{Scalability and centralization}

Papadaki et~al.~\cite{papadaki} highlighted the difficulty of
achieving efficiency for complex aggregate queries at scale.

Kamdar and Musen~\cite{kamdar_empirical_2021} argued for centralized
top-down approaches to monitor accessibility and enforce vocabulary
reuse standards.

rdf-solve's multi-level fallback chain addresses the scalability
concern at the extraction level for external endpoints, recovering patterns for
queries that would otherwise fail under single-shot extraction.
The local mining pipeline (Section~\ref{sec:local-mining}) further
removes the dependency on remote endpoint availability entirely:
downloading data dumps and indexing them with QLever eliminates
network timeouts, rate limits, and engine-specific quirks from the
extraction path.
The containerized web application and the batch-seeding scripts provide a
reusable infrastructure for periodic re-mining, moving toward the
centralized monitoring advocated by Kamdar and Musen~\cite{kamdar_empirical_2021}.

\subsection{Mapping quality and evidence chains}
\label{sec:mapping-quality-disc}

The mapping layer's quality is bound by its upstream sources.
Instance-level mappings from URI probing, SSSOM imports, and SeMRA imports each
inherit the precision and recall of their respective inputs~Hoyt et~al.~\cite{hoyt_assembly_2025}.
Inference propagation (transitivity, inversion, generalisation) amplifies both
signal and noise: transitive chains of length $\geq 3$ carry
multiplicative uncertainty.
The evidence-chain metadata preserved in the mapping JSON-LD
documents these provenance paths, but downstream consumers must
decide which evidence depth to trust.

The current confidence function (Algorithm~\ref{alg:classderive}) is a placeholder
calibrated only to raw instance count: $\mathrm{conf}(n) = \min(1, \log_2(1+n)/10)$.
A probabilistic formulation would instead estimate the posterior probability
$P(t_s \equiv t_o \mid \mathcal{E})$ that a type pair represents a genuine
ontological correspondence given evidence set $\mathcal{E}$.
One natural approach, informed by the SeMRA framework's predicate semantics, would
treat each instance edge as a Bernoulli trial: if the true correspondence probability
is $\theta$, the likelihood of observing $n$ supporting edges out of $N$ total
entity pairs touching those types is $\mathrm{Bin}(n; N, \theta)$, with a
Beta prior on $\theta$ updated per predicate type and evidence justification.
This formulation would additionally allow a consistency check across multiple
independent mapping sources: a type pair supported by sources that systematically
disagree on the predicate (e.g.\ some asserting exact match, others only broad match)
would receive a lower posterior than a pair unanimously supported.
These refinements require calibration data and are left for future work.

The type lookup is limited to \texttt{rdf:type} assertions that are
materialised and visible under the query scope in $G$.
Entities whose type membership is only entailed, or is asserted in a named graph not
covered by the lookup query, will not be resolved and their edges will be silently
dropped during derivation.
Run reports record the fraction of edges skipped for this reason, enabling
targeted diagnosis.

\subsection{Limitations of schema mining}
\label{sec:mining-discussion}

\paragraph*{Graph scope and invisible types.}
Phase~1 enumerates only types asserted via \texttt{rdf:type} and
visible under the chosen graph scope; inferred or unmaterialized
types are silently absent from $C$.
Graph selection is therefore a critical configuration choice.
The whole-endpoint mining variant (no named-graph restriction) with
post-mining service-namespace filtering addresses this partially:
it recovers patterns that would be missed under a strict graph scope,
at the cost of including infrastructure artefacts that must be
stripped.
% Expected: quantify how many additional classes the .all mining recovers

\paragraph*{Remote mining.}
The remote mining step depends on endpoint availability, rate limits, and
engine-specific behavior, and is inherently liable to data incompleteness.
Failure risk for $Q_\mathrm{to}$ grows with instance count and object
cardinality, so the output may be biased toward small, well-indexed
classes.
When all fallback strategies fail, a batch contributes $\emptyset$ to
$S$, indistinguishable from a class with genuinely no patterns of
that type.
Failures are logged but cannot be encoded in $S$ itself.

Different SPARQL engines (Virtuoso, Blazegraph, Fuseki, QLever)
exhibit different timeout behavior, error responses...
The fallback chain mitigates but does not eliminate these differences.

\paragraph*{Property-level mappings.}
The current mapping layer operates at the class level: it links
ontological types across datasets but does not align individual
predicates.
Property-level mappings would enable predicate relaxation, e.g.\
mapping domain-specific relation ontology predicates to broader SIO
predicates, which is a prerequisite for generating simplified
property-graph exports and cross-dataset federated queries that
abstract over predicate heterogeneity.
\paragraph*{Validation feedback loops.}
% After some manual inspection and changes, the SHACL shapes generated from mined schemas could be used to
%validate the source data itself, exposing quality issues that are invisible without a schema.
Feeding validation results back to dataset publishers would close the loop between schema mining and data
improvement~Mihindukulasooriya et~al.~\cite{mihindukulasooriya_rdf_2018}.

\paragraph*{Downstream exports.}
LinkML intermediate representations open a path to SQL table
generation, property-graph exports (Neo4j), and typed Python APIs
(Pydantic), extending the utility of mined schemas beyond the RDF
ecosystem.

\paragraph*{Model Context Protocol integration.}
TogoMCP~\cite{togomcp} has shown that exposing database context through
the Model Context Protocol enables LLM agents to generate accurate SPARQL
without manual prompt engineering.
An MCP server layer over rdf-solve would expose mined schemas,
mappings, and connectivity metadata as tool resources consumable by any
MCP-compatible LLM client, effectively scaling the TogoMCP paradigm from
manually curated MIE files to automatically generated schema artefacts
across the full LSLOD cloud.
Combined with the query composition module already present in rdf-solve,
this would provide an end-to-end path from natural-language questions
through schema-informed query generation to cross-endpoint traversal.
